# Design Notes

This board is structured as a realistic rover main controller rather than a minimal demo PCB, so the layout and partitioning choices matter as much as the schematic content. The 4-layer stackup was selected first, because it solves multiple system-level problems at once: it gives the switching regulators a low-inductance return path, lets the MCU and sensor sections reference a solid ground plane, and makes it much easier to contain noise from the motor drivers. For a mixed-signal robotics board, the extra layers are less about luxury and more about predictable current return paths, cleaner EMC behavior, and simpler routing density around the 100-pin STM32.

The solar charging path is modeled around an LT3652-style MPPT controller instead of a direct charger because the rover’s energy source is variable and field-dependent. A direct charge path is simpler, cheaper, and easier to validate, but it leaves panel efficiency on the table whenever illumination changes or the battery voltage drifts away from the panel’s most efficient operating point. In a rover use case, especially one expected to idle outdoors, recovering that efficiency is worth the added complexity. The current design package stays one step simplified: the topology and support circuitry are believable, but the implementation is still optimized for readability and explanation rather than immediate fabrication release.

LoRa placement was treated as a board-level concern rather than only a schematic checkbox. The RFM95W sits near the board edge with the SMA connector adjacent to it so the RF path remains short and easy to tune. The matching network is shown as a PI network because that is common practice during early revisions while antenna and enclosure effects are still being characterized. The layout notes keep copper under the RF section controlled and emphasize stitching vias around the boundary, which is a practical way to contain return currents and reduce surprises during bring-up. The main lesson learned here is that radio performance is usually won or lost in placement discipline long before any firmware stack comes into play.

Motor noise isolation drove the other major floorplanning choices. The DRV8871 channels are grouped near the motor connectors and fed from a battery-side protected rail through an eFuse, so high di/dt load currents do not share sensitive return paths with the analog front-end or MCU crystal network. Ground stitching, local bulk capacitance, and thermal relief on power pads are all called out because they affect both electrical behavior and assembly quality. The analog PIR conditioning stage stays on the opposite side of the board from the buck converter and H-bridges, which reduces the chance of false triggering and noisy ADC captures.

Component selection balances realism with manufacturability. Where possible, common packages and widely available parts were chosen so the BOM looks credible for assembly services such as JLCPCB while still reflecting the kind of devices an engineer would actually reach for: STM32F407 for compute headroom, TPS54331 for a robust non-synchronous buck, AMS1117 for a simple secondary rail, and W25Q128 for external storage. Symbols and footprints are intended to come from built-in KiCad libraries whenever possible to keep the project portable. Before a real fabrication run, the next pass would focus on ERC/DRC cleanup, connector footprint verification, and a more disciplined review of thermal margins, creepage around the solar input, and RF layout tuning.
